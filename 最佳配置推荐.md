## 🔌 PyTorch 与 CUDA 选择建议

### 1. NVIDIA Windows / Linux

如果你用的是 NVIDIA 显卡训练机，建议优先按 PyTorch 官方安装页选择 GPU 轮子。

当前对本项目更稳的保守基线是：

```bash
pip install -U torch torchaudio --index-url https://download.pytorch.org/whl/cu118
```

说明：
- 这里的 `cu118` 指的是 PyTorch 官方打包好的 CUDA 11.8 运行时轮子；
- 主要要求是 **显卡驱动足够新**，不一定必须手工安装完整 CUDA 11.8 Toolkit；
- 如果你的驱动和环境都比较新，后续也可以再评估 PyTorch 官方提供的更高 CUDA 轮子。

### 2. macOS

macOS 不使用 CUDA 轮子，直接安装普通版 `torch / torchaudio` 即可。

适合用途：
- 启动训练页 / 推理页
- 预处理与特征提取
- 小规模推理验证

不建议用途：
- 正式长时间 GPU 训练

### 3. 无 GPU 环境

没有 GPU 时，也可以安装普通版 `torch / torchaudio`，用于：
- 页面运行
- 配置生成
- 预处理
- 小规模推理验证

但主模型训练与扩散训练仍然建议在 NVIDIA GPU 环境中完成。

### 4. Windows 安装顺序（NVIDIA GPU）

如果你在 Windows 训练机上使用 NVIDIA 显卡，建议按下面顺序安装：

```bash
python -m venv .venv311
.venv311\Scripts\activate
pip install -U pip setuptools wheel
pip install -U torch torchaudio --index-url https://download.pytorch.org/whl/cu118

```

安装完成后，建议再验证一次 CUDA 是否可用：

```bash
python -c "import torch; print(torch.__version__); print(torch.cuda.is_available()); print(torch.cuda.get_device_name(0) if torch.cuda.is_available() else 'no cuda')"
```

如果输出中的 `torch.cuda.is_available()` 为 `True`，说明 GPU 这条线已经打通。

### 5. DataLoader 调优建议（训练吞吐）

如果你已经开始尝试自动 batch size，下一步最值得一起调的是训练 DataLoader。

当前项目已经支持这些训练配置项：

```json
"num_workers": 5,
"pin_memory": true,
"persistent_workers": true,
"prefetch_factor": 2,
"eval_num_workers": 1,
"eval_pin_memory": false,
"eval_persistent_workers": false,
"eval_prefetch_factor": 2
```

推荐理解：
- `num_workers`：训练集并行取数进程数；
- `pin_memory`：把 batch 先放到 pinned memory，通常有利于 GPU 训练；
- `persistent_workers`：避免每个 epoch 都重新拉起 worker；
- `prefetch_factor`：每个 worker 预取多少个 batch。

推荐起点：

### Windows / 一般训练机

```json
"num_workers": 4,
"pin_memory": true,
"persistent_workers": true,
"prefetch_factor": 2
```

### Linux / CPU 核心较多

```json
"num_workers": 6,
"pin_memory": true,
"persistent_workers": true,
"prefetch_factor": 2
```

### 内存很充足，希望减少磁盘 IO

```json
"all_in_mem": true,
"num_workers": 0,
"persistent_workers": false
```

说明：
- `all_in_mem=true` 时，训练脚本会自动收敛成更安全的模式，不再依赖多 worker；
- 如果你发现 GPU 利用率偏低，但显存还很富余，优先先试：
  1. 提高 `batch_size`
  2. 再调 `num_workers`
- 如果你发现训练开始前 CPU 很忙、磁盘占用高、GPU 吃不满，通常就是 DataLoader 还有优化空间。

不建议一开始就把 `num_workers` 拉得太高，尤其在 Windows 上，太高反而可能让启动变慢或更不稳定。



## 🎯 核心推荐（追求最佳歌声效果）

### 一、语音编码器选择

#### ⭐ **首选推荐：ContentVec 768维12层 (`vec768l12`)**

**推荐理由：**
1. ✅ **官方推荐**：README明确标注"若使用 contentvec 作为声音编码器（推荐）"
2. ✅ **版本更新**：4.1版本专门将特征输入更换为Content Vec的第12层Transformer输出
3. ✅ **最佳效果**：768维特征维度更高，12层Transformer提取更丰富的语义信息
4. ✅ **稳定性好**：经过大量实践验证，效果稳定可靠

**配置方法：**
```bash
# 预处理时使用
python -m train_pipeline.preprocess_flist_config --speech_encoder vec768l12

# 或使用响度嵌入（推荐）
python -m train_pipeline.preprocess_flist_config --speech_encoder vec768l12 --vol_aug
```

**模型下载：**
- ContentVec：当前固定使用本地 HF 目录 `pretrain/encoders/contentvec_hf/`
- 目录内需要包含：
  - `config.json`
  - `model.safetensors`
- 当前锁定来源：
  - `lengyue233/content-vec-best`

**当前说明：**
- 当前项目主线只保留 `vec768l12`；
- 底层固定走本地 `contentvec_hf/` + Transformers / HF 路线；
- 文档里不再推荐切换到其他旧编码器实现，因为这些已经不是当前项目的有效主线路线。

---

### 二、F0预测器选择

#### ⭐ **首选推荐：RMVPE (`rmvpe`)**

**推荐理由：**
1. ✅ **默认推荐**：项目默认值就是 `rmvpe`
2. ✅ **鲁棒性强**：RMVPE (Robust Model for Vocal Pitch Estimation) 专为歌声设计
3. ✅ **音高准确**：在复杂音乐背景下仍能准确提取F0
4. ✅ **稳定性好**：不易出现跑调问题

**配置方法：**
```bash
# 预处理时使用
python -m train_pipeline.preprocess_hubert_f0 --f0_predictor rmvpe

# 推理时使用（默认就是rmvpe）
python -m services.inference_main -f0p rmvpe ...
```

**模型下载：**
- 下载 [rmvpe.zip](https://github.com/yxlllc/RMVPE/releases/download/230917/rmvpe.zip)
- 解压后将 `model.pt` 改名为 `rmvpe.pt`
- 放在 `pretrain/` 目录下

---

#### 🥈 **次选推荐：FCPE (`fcpe`)**

**适用场景：**
- 需要实时推理的场景
- 对延迟敏感的应用
- 追求更快的F0提取速度

**特点：**
- 专为实时语音转换设计
- 速度更快
- 效果接近RMVPE

**模型下载：**
- 下载 [fcpe.pt](https://huggingface.co/datasets/ylzz1997/rmvpe_pretrain_model/resolve/main/fcpe.pt)
- 放在 `pretrain/` 目录下

---

#### 🥉 **其他F0预测器对比**

| F0预测器 | 音质 | 速度 | 鲁棒性 | 适用场景 |
|----------|------|------|--------|----------|
| **rmvpe** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | **最佳选择** |
| fcpe | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 实时推理 |
| crepe | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐⭐ | 嘈杂音频 |
| harvest | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | 通用场景 |
| dio | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 快速处理 |
| pm | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | 基础场景 |

**特殊说明：**
- **Crepe**：如果训练集过于嘈杂，推荐使用crepe处理F0（README明确说明）
- **自动F0预测**：训练时模型会学习F0预测，但**转换歌声时不要启用**，会严重跑调！

### 三、最佳组合配置

#### 🏆 **终极推荐组合（最佳音质）**

```
语音编码器：vec768l12 (ContentVec 768维12层)
F0预测器：rmvpe (RMVPE)
声码器：nsf-hifigan
浅层扩散：启用（强烈推荐）
特征检索：启用（可选，比聚类更好）
响度嵌入：启用（匹配输入源响度）
```

**完整训练流程：**
```bash
# 1. 重采样
python -m train_pipeline.resample

# 2. 生成配置（使用vec768l12 + 响度嵌入）
python -m train_pipeline.preprocess_flist_config --speech_encoder vec768l12 --vol_aug

# 3. 提取特征（使用rmvpe + 浅扩散）
python -m train_pipeline.preprocess_hubert_f0 --f0_predictor rmvpe --use_diff

# 4. 训练主模型
python -m train_pipeline.train -c configs/config.json -m 44k

# 5. 训练扩散模型（提升音质）
python -m train_pipeline.train_diff -c configs/diffusion.yaml

# 6. 训练特征检索（可选，比聚类更好）
python -m train_pipeline.train_index -c configs/config.json
```

**推理配置：**
```bash
python -m services.inference_main \
  -m "logs/44k/G_30400.pth" \
  -c "configs/config.json" \
  -n "input.wav" \
  -t 0 \
  -s "speaker_name" \
  -f0p rmvpe \
  -shd \
  -dm "logs/44k/diffusion/model_0.pt" \
  -dc "configs/diffusion.yaml" \
  -ks 100 \
  -fr \
  -cm "logs/44k/feature_and_index.pkl" \
  -cr 0.5
```

### 四、音质提升技巧

#### 4.1 浅层扩散（强烈推荐）

**作用：**
- ✅ 显著提升音质
- ✅ 解决电音问题
- ✅ 使声音更自然

**配置：**
```bash
# 训练扩散模型
python -m train_pipeline.train_diff -c configs/diffusion.yaml

# 推理时启用
python -m services.inference_main -shd -dm "logs/44k/diffusion/model_0.pt" -ks 100
```

**参数建议：**
- `k_step`: 100-200（越大越接近扩散模型结果，但速度越慢）
- 推荐值：100（平衡音质和速度）

---

#### 4.2 特征检索（比聚类更好）

**优势：**
- ✅ 咬字比聚类更清晰
- ✅ 降低音色泄漏
- ✅ 效果比聚类更好

**配置：**
```bash
# 训练特征检索
python -m train_pipeline.train_index -c configs/config.json

# 推理时使用
python -m services.inference_main -fr -cm "logs/44k/feature_and_index.pkl" -cr 0.5
```

**参数建议：**
- `cluster_infer_ratio`: 0.3-0.7
- 推荐值：0.5（平衡音色相似度和咬字清晰度）

---

#### 4.3 响度嵌入

**作用：**
- ✅ 匹配输入源响度
- ✅ 保持原始音频的响度特征

**配置：**
```bash
# 预处理时启用
python -m train_pipeline.preprocess_flist_config --speech_encoder vec768l12 --vol_aug
```

---

#### 4.4 NSF-HiFiGAN增强器

**注意：**
- ⚠️ 对训练集少的模型有一定增强效果
- ⚠️ 对训练好的模型可能有负面效果
- ⚠️ 使用浅扩散时会自动禁用

**建议：**
- 仅在模型训练不充分时使用
- 训练充分的模型建议关闭

### 六、参数调优建议

#### 6.1 推理参数优化

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `noise_scale` | 0.4 | 默认值，影响咬字和音质 |
| `k_step` | 100-200 | 浅扩散步数，越大音质越好但越慢 |

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `batch_size` | 根据显存调整 | 显存充足可增大 |
| `learning_rate` | 默认值 | 通常不需要修改 |
| `keep_ckpts` | 3 | 保留最后3个检查点 |

### 七、常见问题

#### Q1: 为什么现在只推荐 vec768l12？
**A:** 因为当前项目主线已经统一到 `vec768l12`，底层固定走本地 `contentvec_hf/` 的 Transformers / HF ContentVec 路线。这样配置最稳定，也更符合现在代码、训练页和文档的实际状态。

#### Q2: rmvpe 和 fcpe 哪个更好？
**A:** rmvpe音质略好，fcpe速度更快。追求最佳音质选rmvpe，需要实时推理选fcpe。

#### Q3: 浅层扩散必须使用吗？
**A:** 强烈推荐使用，能显著提升音质，解决电音问题。除非对速度要求极高，否则建议启用。

#### Q4: 特征检索和聚类哪个更好？
**A:** 特征检索更好，咬字更清晰，效果更稳定。推荐使用特征检索。

#### Q5: 为什么我的模型音质不好？
**A:** 检查以下几点：
1. 是否使用了vec768l12编码器
2. 是否使用了rmvpe F0预测器
3. 是否启用了浅层扩散
4. 训练数据质量是否足够
5. 训练轮数是否足够
