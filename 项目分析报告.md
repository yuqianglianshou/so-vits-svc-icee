## 二、当前项目定位

当前项目仍然是一个以 **歌声音色转换** 为核心的 So-VITS-SVC 工程，但已经不是单纯的“训练脚本 + 推理脚本”仓库，而是收敛为两个主要入口：

### 2.1 训练入口

- `app_train.py`
- 作用：
  - 检查并导入训练前依赖与底模
  - 管理 `dataset_raw/<说话人目录>/*.wav`
  - 导入单说话人数据
  - 删除当前说话人目录
  - 自动同步处理后目录 `dataset/44k/<说话人目录>`
  - 按单说话人模式分步骤执行重采样、配置生成、特征提取、主模型训练、扩散训练、索引训练
  - 让每个说话人对应独立模型输出目录 `logs/<模型名>/`
  - 提供训练前检查、阶段判断、任务状态、日志尾部、错误/警告提取

### 2.2 推理入口

- `app_infer.py`
- 作用：
  - 加载主模型、音质增强模型、音色增强文件
  - 歌声转换
  - 高音质质量模式切换
  - 运行摘要导出
  - 面向“离线高质量转换”而不是实时低延迟

## 四、当前能力清单

### 4.1 前置检查与文件导入

当前项目已经具备统一主入口能力，不再需要用户先手工把文件塞进固定目录。

#### 已支持

- 预训练依赖检查：
  - `pretrain/encoders/contentvec_hf/`
  - `pretrain/encoders/rmvpe.pt`
  - `pretrain/vocoders/nsf_hifigan/`
  - `pretrain/base_models/44k/G_0.pth`
  - `pretrain/base_models/44k/D_0.pth`
  - `pretrain/base_models/44k/diffusion/model_0.pt`
- 项目模型文件检查：
  - `logs/44k/G_*.pth`
  - `logs/44k/config.json`
  - `logs/44k/diffusion/*.pt`
  - `logs/44k/diffusion/config.yaml`
  - `logs/44k/feature_and_index.pkl`
- 提供下载地址
- 允许从本地选择文件并导入到项目标准位置
- 根据当前文件情况给出“推荐下一步”

### 4.2 训练控制台

相较旧报告，训练侧已经有明显增强。

#### 已支持

- 模型工作区切换 / 新建 / 删除
- 训练语音数据导入到当前模型绑定的数据目录
- 显示当前语音数据目录里的 wav 文件列表
- 删除当前语音数据
- 训练前检查
- 阶段判断
- 一键执行 1-3 步
- 一键执行到主模型训练
- 启动 TensorBoard
- 任务状态、日志尾部、错误/警告提取

#### 当前设计上的关键改进

- 原始语音数据目录与处理后目录已经按当前模型工作区绑定：
  - `dataset_raw/paimeng` -> `dataset/44k/paimeng`
- 每个训练模型对应一个独立模型输出目录：
  - `logs/paimeng/`
  - `logs/ningguang/`
- 避免了不同数据集共享同一个模型目录带来的配置、扩散和索引产物覆盖
- 数据目录名已做基本安全净化，避免越出 `dataset_raw`
- 当前产品口径已经明确为单说话人训练，不再把多说话人混合训练作为默认流程开放

#### 当前训练模式结论

- 底层 So-VITS-SVC 结构仍然保留多说话人能力
- 但当前仓库的训练控制台已经收敛为 **单说话人模式**
- 当前推荐工作流是：
  - 一个说话人目录
  - 一个处理后目录
  - 一个模型输出目录

如果第一个说话人已经训练完成，训练第二个说话人时的正确方式不是把两个说话人混进一个模型，而是：

1. 切换到第二个说话人目录
2. 使用新的模型输出名
3. 让新产物写入新的 `logs/<模型名>/`

示例：

- 第一个说话人：
  - `dataset_raw/paimeng/`
  - `dataset/44k/paimeng/`
  - `logs/paimeng/`
- 第二个说话人：
  - `dataset_raw/ningguang/`
  - `dataset/44k/ningguang/`
  - `logs/ningguang/`

### 4.3 推理工作台

推理页现在更偏“高质量工具”，不再强调实验性入口。

#### 已支持

- 上传或本地选择模型
- 可选加载音质增强模型和音色增强文件
- 质量模式切换
  - 标准高质
  - 极致质量
- 统一高音质预设
- 生成运行摘要
- 文件完整度提示
- 结果播放器与输出路径反馈

#### 当前取向

- 默认流程面向 **离线高质量歌声转换**
- 不以实时性为优先目标
- 参数收敛到尽量少的关键项，减少小白误操作

### 4.4 稳定性修复

近期已经修过几类高优先级问题：

- `Svc.infer()` 返回值与调用方不一致
- 说话人 ID 解析不稳
- DDP 训练未使用 `DistributedSampler`
- 预处理多进程在 `spawn` 下传不可 pickle 对象
- 推理重采样器判断逻辑引用错误对象
- Gradio 4 升级后首页 API schema 生成不稳定

其中最后一项现在的处理方式是：

- 主路径默认不启用兼容补丁
- 只在需要时通过 `GRADIO_ENABLE_API_INFO_PATCH=1` 显式启用
- 兼容文件为 `gradio_api_info_fallback.py`

这意味着当前项目已经尽量贴近 Gradio 4 默认行为，兼容层只作为兜底保留。

这些修复已经明显高于旧报告里的“泛泛建议”，属于已经落地的工程修正。

## 五、旧报告与当前项目的差异

下面是旧报告中最明显的过时点。

### 5.1 入口文件已过时

旧报告仍使用：

- `main_portal.py`
- `train_webUI.py`
- `webUI.py`

当前应更新为：

- `app_train.py`
- `app_infer.py`

### 5.2 UI 结构已变化

旧报告把 Web 界面视为单一入口，这已经不准确。当前是：

- 训练页负责训练前依赖和训练流程
- 推理页负责转换流程

这两者已经分工。

### 5.3 旧报告高估了“实验功能”的重要性

旧报告强调：

- 实时推理
- 实验室工具
- 文字转音频
- 模型压缩工具在 UI 中的展示

但当前项目的产品方向已经明确：

- 以高质量离线歌声转换为主
- 训练和推理页面尽量去掉分散注意力的功能

所以旧报告里那些“继续扩 UI 功能”的建议，不应该照单全收。

### 5.4 旧报告没有覆盖当前真正的工程问题

例如：

- 训练页数据集目录污染风险
- 目录名输入安全问题
- 主入口与子页面的进程管理
- 统一高音质预设
- 页面职责拆分

这些都是当前项目实际遇到并已经处理或正在处理的问题，旧报告没有触达。

建议：

- `app_train.py`
- `app_infer.py`
  作为正式入口

### 7.2 Windows GPU 阶段的高优先级验证点

这些是下一阶段真正重要的。

#### 7.2.1 训练链路闭环

- 导入说话人数据
- 重采样
- 配置生成
- 特征提取

重点听感项：

- 音准

- 音色相似度

Mac 本地只能看界面和做轻量交互，真正要确认的是：

- 用户是否能在 Windows 上顺利导入文件
- 浏览器上传目录的行为是否稳定
- 训练/推理启动后的日志与提示是否足够明确

### 7.3 训练吞吐优化第一阶段（已完成）

这一轮已经不只是“分析速度问题”，而是已经落下了第一阶段的结构性优化。

#### 7.3.1 已完成的点

1. DataLoader 调优入口
- 主模型训练模板已支持：
  - `num_workers`
  - `pin_memory`
  - `persistent_workers`
  - `prefetch_factor`
  - 以及 eval 对应配置
- 主模型训练脚本已按配置读取这些参数

2. 主模型训练数据组织优化第一阶段
- 预处理第 3 步会额外生成：
  - `*.train.pt`
- 当前统一特征包已包含：
  - `audio`
  - `soft`
  - `f0`
  - `uv`
  - `spec`
  - `volume`
- 主模型训练当前优先并实际依赖：
  - `*.train.pt`
- 训练页第 3 步状态也已纳入：
  - `trainpt=<数量>`

#### 7.3.2 这一轮优化的实际意义

当前主模型训练的数据读取路径，已经从原先的：

- `wav`
- `.spec.pt`
- `.f0.npy`
- `.soft.pt`
- `.vol.npy`

进一步收紧到：

- **主路径依赖 `*.train.pt`**

这意味着：

- 小文件读取次数减少
- 训练时不再优先碰原始 `wav`
- 主模型训练的读取链条已经明显比旧结构更紧凑

#### 7.3.3 当前刻意没有继续动的点

这一轮没有继续把扩散训练也并进统一打包。

原因：

- 扩散训练除了共享的 `soft / f0 / volume` 之外，
- 还强依赖：
  - `mel`
  - `aug_mel`
  - `aug_vol`
- 并且当前扩散训练有自己独立的缓存策略：
  - `load_all_data`
  - `data_buffer`

所以当前更合理的策略是：

- **先把主模型训练主链收紧**
- **扩散训练的数据组织优化单独评估**

#### 7.3.4 下一步真正需要验证的，不是继续改代码

当前更关键的是在 Windows + NVIDIA GPU 真机上确认：

1. 自动 batch size 探测是否能给出稳定建议值
2. DataLoader 调优后 GPU 是否更容易吃满
3. `*.train.pt` 主路径是否能带来可观察的训练吞吐改善

也就是说，训练吞吐优化这条线现在已经进入：

- **代码第一阶段已落地**
- **下一步靠真机验证收益**

## 八、结论

### 8.1 旧报告是否还值得保留

值得保留，但只能作为“项目背景材料”，不能继续当作当前工程状态说明。

原因：

- 基础技术判断大体没错
- 但入口、页面职责、工程优先级已经变了

### 8.2 旧建议是否还要照做

不能照单全收。

应按下面三类处理：

- 已完成或已部分完成：
  - 界面优化
  - 参数预设
  - 依赖收敛
  - 错误提示增强
  - 部分回归检查

- 仍值得做：
  - 最小测试体系
  - 输入校验增强
  - 日志统一
  - README / 文档进一步统一

- 现在不应优先：
  - 实时推理强化
  - ONNX / TensorRT 加速
  - 模型版本管理系统
  - 大规模类型提示与注释工程

### 8.3 当前建议的工作顺序

1. 在当前仓库状态下提交 GitHub
2. 到 Windows + GPU 环境验证训练与推理闭环
3. 根据真实问题修训练链路与质量参数
4. 再补最小自动化测试和日志统一
5. 最后再考虑更大规模的工程化整理

这比继续按旧报告做“泛化优化”更合理，也更符合当前项目目标。
